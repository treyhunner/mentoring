Teach, don't tell

"You learn by doing"

- So ... ... - I often live code when I teach - And I'm always tempted to *write code out* and then *explain* it, but that's not teaching **(click)**... that's telling. **(click)** - *teach, don't tell* **(click)** - You can do that by encouraging **active recall**: that's the act of retrieving information from your head - ... - We **don't learn** by trying to *put information into our heads* - We *learn* by attempting to retrieve information *from* our heads - *(pause)* - Who has heard the expression "you learn by doing"? - *(pause)* It's true. You don't learn by consuming information, you learn by retrieving it: the doing - ... - This is tricky because *the doing* is **intimidating** when you don't know what you're doing. - You can help your learner by **encouraging them**...

Rubber duck debugging

  • "My code doesn't work"
  • "What do you expect it to do?"
  • "What does it do instead?"
  • "What approach are you taking?"
- With rubber duck debugging, when you get stuck on a coding problem you talk to you rubber duck - Your rubber duck doesn't talk back to you, it just listens - So the duck doesn't answer any questions, but talking to the duck helps you elaborate on your problem which can help you uncover ideas you hadn't thought of yet - I want you to do a lot more listening to your learner, than talking - So if your learner says **(click)** "my code doesn't work" you could ask **(click)** what they expect it to do and**(click)** what it does instead and then follow that up by asking them **(click)** to explain the approach they're taking - Elaboration is a great way to learn

Speak Deliberately

  1. "What does this do?"
  2. "What will happen when we run this?"
  3. "What do you think will happen when we run this?"
  4. "What do you think might happen when we run this? What's your guess?"
- Here are 4 ways to ask almost the same thing... - **4 Clicks** - Which of these is the most welcoming? - I would say number 4. - Those first and second ones sound kind of like a **pop quiz** - Active recall is how we learn: so you *do* want to quiz your learner, but you don't want them to **feel like they're being quizzed** - You want to invite your learners *to guess* and *to make mistakes* by using welcoming language, not confident or assertive language - So I want you to think less about what you're *saying* and more about *what might be heard* - And I want you to be **encouraging**

Encourage, don't alienate

  • "So obviously, ..."
  • "Of course, ..."
  • "Oh it's easy, you just... "
- There are a lot of ways you can alienate you learner, in big ways and in small ways through the words you use - "So obviously" **(click)**: many things are **not obvious** when you're learning - "Of course" **(click)**: when you're new to something, there aren't many "of course" situations - "It's easy, you just" **(click)**: it's easy for you, it's not easy if you don't know it yet - What's in your head as a coach, is not in your learner's head - What's easy for you, is not easy for your learner. It's hard. - Your learner doesn't yet know the same things you do, because you're further along in the process of learning - ... *(breathe)* - ... - This next slide is the scariest one I have

Take things one step at a time

  1. A non-existent (misspelled) function being called
  2. Some code that's poorly styled, but syntactically valid
  3. An error in understanding of the technical requirements
  4. A syntax error that's stopping the code from running
- Let's say your learner says: "my code doesn't work" - They show you the code, and you see 4 things: - **(click)** A function that's improperly named - **(click)** Code that's a little ugly, but it works - **(click)** Something that seems like a logical error: a misunderstanding of the problem you're solving - **(click)** And a syntax error: something that stops the code from running at all - Which one of these do you focus on? - **Trick question**. The answer is actually whichever one is currently most important to your learner - Unless they're asking you about code style, that's not the most pressing issue of these four **(*elaborate*)** **(click)** - The most important *right now* might be the one Python is *complaining* about: the syntax error **(click)** - You don't want to distract your learner: so just focus on *the one thing* that's important *right now*. And then stop. *The rest* can come later. - TODO one thing priority visual then stop sign visual?
