Teach, don't tell

"You learn by doing"

- I often live code when I teach - And I'm always tempted to *write code out* and then *explain* it, but that's not teaching **(click)**... that's telling. **(click)** - *teach, don't tell* **(click)** - You can do that by encouraging your learners to practice **information retrieval** - We don't learn by trying to *put information into our heads* - We *learn* by attempting to retrieve information *from* our heads - *(pause)* - Who has heard the expression "you learn by doing" **(click)**... you really *do* learn by *doing*: you learn by trying to take information out of your head and apply it - Which is tricky because when you're new to something, the thing you're **not yet good at** is *the doing* - You can make this problem of **not knowing** a little less intimidating by **encouraging your learner to make mistakes**

Encourage making mistakes

Speak Deliberately

  1. "What does this do?"
  2. "What will happen when we run this?"
  3. "What do you think will happen when we run this?"
  4. "What do you think might happen when we run this? What's your guess?"
- Here's 4 ways to say almost the same thing... - **4 Clicks** - Which of these is the most welcoming? I would say number 4. - Those first and second ones sound kind of like a pop quiz - You *do* want to encourage information retrieval, but you don't want to make it **feel like a quiz**: you want to invite your learners *to guess* and *to make mistakes* - Confident and assertive language is often counter-productive when you're coaching someone through a problem - So I want you to think less about what you're saying and more about *what might be heard* - And if possible, I want you to be **encouraging**

Take things one step at a time

  1. A non-existent (misspelled) function being called
  2. Some code that's poorly styled, but syntactically valid
  3. An error in understanding of the technical requirements
  4. A syntax error that's stopping the code from running at all
- Let's say your learner says: "my code doesn't work" - They show you the code, and you see 4 things: - **(click)** A function that's improperly named - **(click)** Code that's a little ugly, but it works - **(click)** Something that seems like a logical error: a misunderstanding in the actual problem you're solving - **(click)** And a syntax error: something that stops the code from running at all - Which one of these do you focus on? - The answer is whichever one is currently most important to your learner - Unless they're asking you about code style, that's not the most pressing issue of these four: that's a discussion *for some other time* - The most important *right now* is often the one that Python is *complaining* about: which is probably the syntax error **(click)** - You don't want to distract your learner: so just focus on *the one thing* that's important *right now*. And then stop. *The rest* can come later. - TODO one thing priority visual then stop sign visual?
